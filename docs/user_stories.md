# User Stories

## Роли в системе

| Роль | Описание |
|------|----------|
| **Сотрудник приёмной комиссии** | Обрабатывает заявки, меняет статусы |
| **Аналитик** | Просматривает метрики, выгружает отчёты |
| **Руководитель** | Следит за ходом кампании на уровне факультетов |

## Приёмная комиссия

| ID | User Story | Приоритет |
|----|-----------|-----------|
| US-01 | Как сотрудник приёмной комиссии, я хочу видеть список всех заявок с фильтрами по источнику, волне и статусу, чтобы быстро найти нужную заявку | Must |
| US-02 | Как сотрудник приёмной комиссии, я хочу открыть карточку заявки и увидеть данные абитуриента и историю смены статусов, чтобы понять на каком этапе находится заявка | Must |
| US-03 | Как сотрудник приёмной комиссии, я хочу изменить статус заявки с new на review или enrolled / rejected, чтобы отразить реальное состояние рассмотрения | Must |
| US-04 | Как сотрудник приёмной комиссии, я хочу получать ошибку при попытке изменить терминальный статус (enrolled / rejected), чтобы случайно не испортить данные | Must |
| US-05 | Как сотрудник приёмной комиссии, я хочу видеть лог всех изменений статуса с указанием времени и имени сотрудника, чтобы отследить кто и когда менял заявку | Should |

## Диаграмма статусов заявки

```
         взят в работу          зачислен
  new ──────────────────► review ──────────► enrolled  (терминальный)
   │                         │
   │   прямой отказ          │ отказ
   └─────────────────────────┴──────────────► rejected  (терминальный)
```

**Правила переходов:**
- `new` → `review` — заявка взята в работу сотрудником
- `review` → `enrolled` — абитуриент зачислен
- `review` → `rejected` — абитуриенту отказано
- `new` → `rejected` — прямой отказ без рассмотрения (редкий случай)
- Из `enrolled` и `rejected` переход невозможен — это терминальные статусы
